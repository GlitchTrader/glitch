## Code findings (wave-2 refresh pipeline)

### RP-1 · Unprotected dispatcher apply — fix recommended (P1)

`GlitchMainWindow.RefreshPipeline.partial.cs` → `MarshalAccountRefreshResult`: the `Dispatcher.BeginInvoke` action wraps its body in `try/finally` with **no catch**. `ApplyFullAccountRefreshResult` runs `ApplyAccountRows`, `ApplyRiskMitigations`, `ExecuteReplicationCycle`, `PublishGlitchShellState`, and tab refreshes — all of which previously executed inside `OnRefreshTimerTick`'s top-level catch (`RecordSubsystemFault`). Now any exception in that path escapes to the WPF dispatcher; the AddOn registers no `DispatcherUnhandledException` handler (grep-verified), so a transient replication/risk exception escalates from "quiet subsystem fault" to a potential NT8 crash.

**Fix:** wrap the body (before `finally`) in `try/catch` → `RecordSubsystemFault("account_refresh", ex)`. Small, safe, restores the PA-4 degrade doctrine.

### RP-2 · Replication cycle scheduling under the new pipeline — verify in LANE-1

Heavy ticks no longer run `ExecuteReplicationCycle` synchronously; it runs inside the background-built apply (Background dispatcher priority), while 500ms light ticks still run it synchronously. Consequences: (a) the cycle can fire twice within <500ms (apply + next light tick); (b) on a heavy tick, replication timing depends on thread-pool + dispatcher scheduling — exactly when the UI is busiest. Submit cooldowns should make double-fire safe, but the queued LANE-1 replication audit must verify idempotency under this pipeline explicitly. Alternative worth considering: run replication only on the synchronous tick path and drop it from the deferred apply.

### RP-3 · Stale-display edge cases (minor)

- `UpdateHeaderMetricsFromRows` early-returns when `rows` is empty → header PnL/risk freeze at last values if all accounts disconnect.
- `RefreshAccountDataLight` early-returns when no active accounts → skips `PublishGlitchShellState`, so Chart Trader widgets can hold stale replication state after the last account drops.

### RP-4 · Notes (no action)

- `_accountRefreshBuildInFlight` never resets if the dispatcher drops the queued action (shutdown race) — benign while `_isWindowClosed` gates all entry points.
- Ledger background flush (`GlitchTradeLedgerService` / `GlitchRiskLockLedgerService`): single-flight CAS + synchronous force-flush on close is correct; the bare `catch { }` swallows I/O faults — consider routing to `RecordSubsystemFault` in a later pass.
- Worker-thread safety of `BuildAccountRow` verified: `_selectionOverrides` writes correctly deferred to the UI apply via `deferredAutoOverrides`; `_firmRules` is load-once (constructor only); peak/trade-source maps are `ConcurrentDictionary`. Sequence/coalesce logic is single-UI-thread and correct.
