## A. NinjaTrader 8 current developer guidance

### A1. Order / Execution / Position event ordering guarantees

- `OnOrderUpdate`, `OnExecutionUpdate`, and `OnPositionUpdate` arrive from **non-instrument threads** — live, these are Connection Adapter threads, not the strategy/instrument thread. Each event type is processed sequentially (one `OrderUpdate` fully processed before the next), and each order follows its `OrderState` sequence (Submitted → Accepted → Working → Filled, etc.) in order. `OnExecutionUpdate` is typically raised after the corresponding `OnOrderUpdate`, but this is a *typical* pattern, not a documented cross-event contract. Source: [NinjaTrader Support Forum — OnOrderUpdate Event Chronology](https://forum.ninjatrader.com/forum/ninjatrader-8/add-on-development/1302744-onorderupdate-event-chronology-and-order-cancellation-timing-in-addon-development), [OnExecutionUpdate() — Strategy](https://ninjatrader.com/support/helpguides/nt8/onexecutionupdate.htm), [When is OnPositionUpdate called?](https://ninjatrader.com/support/forum/forum/ninjatrader-8/strategy-development/1121375-when-is-onpositionupdate-called#post1121375).
- **Explicit non-guarantee for Rithmic and Interactive Brokers**: "the sequence of events are not guaranteed due to provider API design" — with multiple simultaneous fills on the same instrument, this can produce an incorrect intermediate `OnPositionUpdate`. This directly matters for Glitch: any account inferred as Rithmic-backed (see `ExecutionProviderHint` in `GlitchComplianceEngine.cs`) cannot assume ordered event delivery. Source: [NinjaTrader Support Forum — OnOrderUpdate Event Chronology](https://forum.ninjatrader.com/forum/ninjatrader-8/add-on-development/1302744-onorderupdate-event-chronology-and-order-cancellation-timing-in-addon-development).
- Order object state should be read from the **event's state parameter**, not the order object's current property, when you need the state *at the time of that event* — the object itself may already reflect a later state due to thread timing. Source: [Multi-Thread Considerations for NinjaScript](https://ninjatrader.com/support/helpguides/nt8/multi-threading.htm).
- **This directly corroborates LANE-1's "OrderUpdate/ExecutionUpdate ordering not guaranteed" concern** noted in the lane briefs — confirmed as accurate per current NT documentation, not merely a hypothesis.

### A2. Multi-account order submission best practices

- Official guidance (NinjaTrader support): *"When using the AddOn Approach, it is best practice to use `CreateOrder()` to create an order"* — i.e. `Account.CreateOrder()` then `Account.Submit()`, not `new Order(...)`. With `CreateOrder()`, the order is immediately registered in NT's internal database; with `new Order()`, it's only registered after submission, making it harder to clean up if submission fails. Source: [AddOn Order Creation and Submission — NinjaTrader Support Forum](https://forum.ninjatrader.com/forum/ninjatrader-8/add-on-development/1186121-addon-order-creation-and-submission).
- For genuinely simultaneous multi-account submission (not one account triggering the next), the documented pattern is: subscribe to account-level `OrderUpdate` on the master, and on receipt, call `Account.CreateOrder()` + `Account.Submit()` per follower account — this is inherently sequential/reactive, not atomic-parallel, which is a latency source worth measuring (relevant to LANE-1's drift/latency finding). Source: [How to access multiple accounts in strategy? — NinjaTrader Support Forum](https://forum.ninjatrader.com/forum/ninjatrader-8/strategy-development/1340402-how-to-access-multiple-accounts-in-strategy).
- **CME regulatory requirement — `OrderEntry.Automated` flag**: this is *policed by the CME*, not just an NT convention. There are documented cases of the CME suspending an account because an AddOn placed automated trades without the flag set to `Automated`. Any order Glitch submits programmatically on behalf of a user (master or follower) should verify `OrderEntry` is set correctly. Source: [Proper Use of OrderEntry — NinjaTrader Support Forum](https://forum.ninjatrader.com/forum/ninjatrader-8/add-on-development/1110891-proper-use-of-orderentry/page2).
- Third-party guidance (CrossTrade, a commercial multi-account tool vendor, not NinjaTrader itself) recommends: every account must hold an active connection at signal time (a disconnected follower simply misses the order — no queueing), and account names must match NT's display exactly (case/spacing) when addressing multiple accounts by name. Source: [CrossTrade — Submit Orders to Multiple NinjaTrader Accounts](https://crosstrade.io/blog/submit-orders-to-multiple-ninjatrader-accounts) (vendor content, treat as corroborating not authoritative).

### A3. `Account.Get(AccountItem.*)` semantics for realized/unrealized PnL

- Syntax: `account.Get(AccountItem.RealizedProfitLoss, Currency.UsDollar)` / `account.Get(AccountItem.UnrealizedProfitLoss, Currency.UsDollar)`. Source: [Total Account Realized and Unrealized PnL — NinjaTrader Support Forum](https://forum.ninjatrader.com/forum/ninjatrader-7/strategy-development-aa/1038161-total-account-realized-and-unrealized-pnl-with-commissions) (NT7 forum but semantics carried into NT8 per cross-references), [Clarification of AccountItem Object](https://forum.ninjatrader.com/forum/ninjatrader-8/strategy-development/92584-clarification-of-accountitem-object).
- `AccountItem.RealizedProfitLoss` is **net of commissions**; the distinct concept "gross realized" P&L does **not** subtract commissions/fees — confirms these are two different figures and a dashboard must be explicit about which one it's showing (directly relevant to LANE-2's math audit / GL-005). Source: [Clarification of AccountItem Object — NinjaTrader Support Forum](https://forum.ninjatrader.com/forum/ninjatrader-8/strategy-development/92584-clarification-of-accountitem-object).
- `AccountItem.CashValue` reflects the broker-reported cash value **including brokerage fees assessed** — not the same denominator as realized/unrealized P&L. Source: same thread.
- `OnAccountItemUpdate` fires for **every account-level item update for the account the script is attached to**, not scoped to the strategy's own orders — a script must filter `AccountItem` type in the handler, it isn't pre-filtered by NT. This is relevant to any Glitch component subscribing to account items expecting only its own trades' effects. Source: [Getting realized profit in an indicator — NinjaTrader Support Forum](https://forum.ninjatrader.com/forum/ninjatrader-8/indicator-development/1327188-getting-realized-profit-in-an-indicator-attached-to-ma).
- I could not find an official NT statement on exactly which `AccountItem` values back the on-screen "Cash Value" / "Realized" / "Unrealized" columns in the native Account Performance / Control Center display pixel-for-pixel — the forum threads describe the API surface, not a guaranteed 1:1 mapping to every UI label. **Mark as UNVERIFIED**: assume `RealizedProfitLoss`/`UnrealizedProfitLoss` match NT's own UI net-of-commission display, but this should be confirmed empirically (which is exactly what LANE-1's repro harness is for) rather than assumed from docs alone.

### A5. AddOn threading / UI-thread rules

- NinjaTrader runs **multiple UI threads plus background worker threads**; the thread that any given NinjaScript/AddOn object executes on is *not* the same thread that raises the triggering event (market data, order/account events, etc.). Source: [Multi-Thread Considerations for NinjaScript](https://ninjatrader.com/support/helpguides/nt8/multi-threading.htm).
- Any UI mutation from a non-UI thread must go through `Dispatcher.InvokeAsync()` on the *correct* dispatcher for that window (e.g. `ControlCenter.Dispatcher.InvokeAsync()` for Control Center, or the specific `ChartControl.Dispatcher` for a chart window) — using the wrong dispatcher, or none, is the standard cause of cross-thread WPF exceptions.
- **Do not use the synchronous `Dispatcher.Invoke()`** — official guidance explicitly warns this "can potentially result in deadlock scenarios." Always prefer `InvokeAsync()`.
- Use `Dispatcher.CheckAccess()` to test whether you're already on the target thread before paying the dispatch cost.
- Shared static state / file access needs explicit locking — the docs cite the classic "file...being used by another process" error as the direct symptom of missing locks around concurrent AddOn file I/O.
- Sources: [Multi-Thread Considerations for NinjaScript](https://ninjatrader.com/support/helpguides/nt8/multi-threading.htm), [Problem with Dispatcher.InvokeAsync](https://forum.ninjatrader.com/forum/ninjatrader-8/add-on-development/1186641-problem-with-dispatcher-invokeasync), [InvokeAsync and Invoke for Dispatcher](https://forum.ninjatrader.com/forum/ninjatrader-8/add-on-development/1318379-invokeasync-and-invoke-for-dispatcher).
- This directly corroborates the "UI-thread marshaling, timers" concern flagged for LANE-1's replication-drift audit — any place in `GlitchMainWindow.Replication.partial.cs` or `GlitchShellBridge.cs` that touches UI state off-thread, or uses synchronous `Dispatcher.Invoke`, is a documented deadlock/exception risk per NT's own guidance, not a hypothetical.

### B4. FundingTicks

- **CLOSED.** Announced wind-down 2026-01-18, support ended 2026-01-31. Source: [Finance Magnates](https://www.financemagnates.com/forex/prop-firm-fundingticks-faces-massive-backlash-after-retroactive-rule-change/).
