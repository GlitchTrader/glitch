### WO-A1 · GL-020 — catch in the marshaled refresh apply (P1, do first)

File: `ninjatrader/Glitch/AddOns/GlitchAddOn/UI/MainWindow/GlitchMainWindow.RefreshPipeline.partial.cs`, method `MarshalAccountRefreshResult`.

The `Dispatcher.BeginInvoke` action currently wraps its body in `try { … } finally { … }`. Exceptions thrown by `ApplyFullAccountRefreshResult` (which runs `ApplyAccountRows`, `ApplyRiskMitigations`, `ExecuteReplicationCycle`, `PublishGlitchShellState`, tab refreshes) escape to the WPF dispatcher — the AddOn registers no `DispatcherUnhandledException` handler, so this can take NT down.

Change: add a `catch (Exception ex) { RecordSubsystemFault("account_refresh", ex); }` between the existing try body and the finally. Nothing else. The finally (in-flight release + coalesce re-queue) must stay exactly as is. `RecordSubsystemFault` already exists in `GlitchMainWindow.Performance.partial.cs` (PA-4) and its degrade path already routes `account_refresh` to the synchronous refresh fallback — that is the designed behavior, don't re-wire it.

Acceptance (for Alan post-compile): temporarily throw inside `ApplyFullAccountRefreshResult` (debug-only line, removed before commit — describe the test in the change doc instead of committing it), confirm NT stays up and a `PerfSubsystemDegraded|account_refresh` Notice appears after 12 faults/min.

### WO-A4 · GL-024 — F1: journal PnL must be net (commission truth)

This is the wave's centerpiece. Read `docs/ledger/audits/pnl-math-audit.md` F1 in full before touching anything.

Problem: dashboard tiles read NT account items (net of commissions when a commission template is applied) while Journal/Analytics recompute **gross** from `PnlPoints × pointValue`. The word "commission" appears zero times in the AddOn. Sim accounts without commission templates mask the bug; funded accounts expose it. North-star: what Glitch displays must equal what NinjaTrader reports.

Architect's chosen seam (option b from the audit — per-trade granularity must survive):

1. **Capture commissions per execution.** Where the trade ledger / insights service consumes NT execution or trade events to build round-trips (start at `GlitchTradeLedgerService` + `GlitchTradeInsightsService`, `Services/Insights/`), read the commission off the NT execution object (`Execution.Commission` — verify the exact property against how neighboring code reads executions; if executions aren't currently consumed and round-trips come from another source, STOP and write the collision note — do not guess a data path).
2. **Store per-round-trip:** `CommissionTotal` (sum over the round-trip's executions) on the ledger trade row. Extend the ledger row model + TSV/file serialization the same way existing fields are serialized; bump whatever row-version marker the file format uses (inspect `GlitchTradeLedgerService` load/save for how it tolerates old rows — old rows read as CommissionTotal=0).
3. **Display net:** Journal totals, Trader Performance table, Analytics aggregates switch to `NetPnl = GrossPnl − CommissionTotal`. Keep gross available as a secondary column/tooltip where the UI already has room (Trader Performance tooltip: "gross X − commissions Y"). All new strings localized.
4. **Reconciliation Notice (quiet, calm-by-default):** once per session per account, compare Journal net total against the dashboard tile source (NT account item realized PnL). If they diverge by more than $1.00, emit ONE Notice (not Critical, no red): `journal_reconcile_divergence|account={a}|journal={x}|nt={y}`. This is our tripwire that the seam is still leaky.
5. **Sim fallback:** accounts with no commission data simply produce CommissionTotal=0 — net equals gross, no notice, no special casing.

Acceptance (Alan): sim session WITH a commission template applied in NT → Journal net total equals NT's realized PnL for the session to the cent; without a template → unchanged behavior. Both runs: zero red elements.
